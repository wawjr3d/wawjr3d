An isomorphic javascript framework

Why did we go isomorphic?

  • Write one language
  • One codebase
  • More options (traditional or SPA)
  • Media sites need SEO
SEO - twitter cards, meta tags, facebook share mission critical

What makes Brisket so good?

  • Code freedom (seed framework)
  • Feels like "just Backbone"
  • Provides conveniences
  • Provides tools to make your decisions
- highly flexible/customizable - helps initialize routers - tools for example normalized request with data like referrer, user agent

Key Concepts/Goals

  • Optimize perceived speed
  • The site works BEFORE javascript starts
  • Feel like a library
- perceived speed is how fast does it take for page to be actionable - read, scroll, navigate - most noticeable and important on a mobile device. initial page may be there but javascript has not begun. want to be able to use the site and have javascript layer in when it's ready - Client side javascript is progressive enhancement - Minimize framework debugging. as much as possible, errors should live in your code where you can find it

How does Brisket actually work?

- 14 MINUTES SHOULD HAVE PASSED. THIS SECTION (8 slides) IS 8 MINUTES. FINAL SECTION IS 3 MINUTES

Basics

  • Provides an express engine
  • CommonJS code bundled for client
  • jsdom on server side
  • Base tag
  • Client app start script
- write your code in CommonJS. use your favorite bundler to prepare for client. must expose a requirable ClientApp - jsdom powers the server side version of jquery - express engine to use in your express server - base tag makes it possible to mount application anywhere - brisket adds a clientapp start script that exposes environment config

Routers return View (or Promise of a View)

A Router's job is to prepare a View to render into Layout

Delegate rendering management to Brisket

ChildView management

Brisket Views can have child views

Brisket child views can reattach themselves to existing DOM

Idempotent child view creation from server to client

Initial page load

Want an instant replay of server side route handling - Sean MacEntee

Initial load state must be same as server for correct rendering

on initial load, we execute the same route that fired on the server. important because when Views "reanimate" on the client, you want them to pick up where they left off.

View reattachment

During server side render, Views get unique identifier

During client side render, Views gets the same unique identifier

Views search for themselves in the DOM
- Tambako The Jaguar

Bootstrapped Data

Server-side API responses are recorded and sent to client

- spy on all jquery.ajax calls. we use node domains to store all api responses for a request. - only successful calls are stored. results in auto-retry logic on client
